너는 이제부터 **Circle(USDC 발행사) 관련 실시간 모니터링/알림 봇**을 만드는 시니어 엔지니어다.
목표는 **Google Cloud VM에서 24시간 상시 실행**되면서, **주가에 영향을 줄 수 있는 새로운 정보가 생겼을 때만** 나에게 즉시 알려주는 봇을 만드는 것이다.

## 1. 최종 목표

다음과 관련된 **새로운 데이터**가 발생하면 검색/수집/판별해서 나에게 알림을 보내는 봇을 만들어라.

### 모니터링 대상

#### A. Circle 직접 관련

* Circle 관련 공시
* 지분공시
* SEC 관련 문서
* 투자/자금조달/상장/IPO 관련 소식
* 파트너십
* 규제 이슈
* 소송, 조사, 제재, 사건, 해킹, 장애, 서비스 중단
* 경영진 발언, 인터뷰, 공식 발표
* 실적, 재무, 사업전략, 신규 상품, 정책 변경
* USDC 준비금, 발행량, 상환, 회계/감사 관련 변화

#### B. 경쟁사 / 비교 대상 관련

* Tether(테더) 관련 뉴스, 공시, 규제 이슈, 사건, 준비금, 감사, 법적 문제
* 기타 스테이블코인 경쟁사(예: Paxos, PayPal stablecoin, Ripple stablecoin 등) 관련 주요 변화
* Coinbase 등 Circle과 실질적으로 연결된 핵심 파트너/유통 관련 중요 뉴스

#### C. 법안 / 제도 / 규제 관련

* 미국 스테이블코인 규제 관련 법안
* GENIUS Act / 지니어스 법안
* Clarity Act / 클래리티 법안
* 스테이블코인, 디지털자산, 결제, 은행 규제, SEC/CFTC/OCC/Fed/재무부 관련 발표
* 의회 발언, 표결 일정, 통과/보류/수정/부결 등 입법 이벤트
* Circle 또는 USDC 사업모델에 영향을 줄 수 있는 주요 제도 변화

#### D. 시장 영향 정보

* Circle, USDC, stablecoin sector 전반에 직접 영향을 줄 수 있는 뉴스
* 대형 거래소 상장/제휴/중단
* 규제기관의 공식 발표
* 경쟁사의 위기/호재
* 시장 신뢰도, 준비금, 감사, 은행 파트너, 유동성 관련 이슈

## 2. 가장 중요한 규칙

### 반드시 "새로운 정보"만 보내라

* **과거에 이미 나온 뉴스/공시/이벤트는 다시 보내지 마라**
* 봇 최초 실행 시에도 과거 뉴스 대량 전송 금지
* 오직 **직전 모니터링 이후 새롭게 확인된 정보만** 보내라
* 같은 내용을 제목만 바꿔서 재배포한 기사도 가능하면 **중복 제거**하라
* 동일 사건을 여러 매체가 보도한 경우:

  * 가능하면 대표 링크 1~3개만 첨부
  * 핵심 사실은 하나로 묶어서 전송
* 업데이트 기사라면:

  * 정말로 **새로운 팩트가 추가된 경우에만** 다시 전송
  * 단순 재탕 기사, 요약 기사, 해설 기사는 제외

### 시간 기준

* **모든 시간은 한국시간(KST, Asia/Seoul) 기준으로 처리**
* 알림 메시지에도 KST 기준 시각 표시

### 링크 첨부 필수

* 모든 알림에는 반드시 **원문 링크** 첨부
* 가능하면 공식 출처 우선:

  * 기업 공식 홈페이지
  * SEC
  * 미 의회/정부 사이트
  * 규제기관 공식 발표
  * 신뢰도 높은 주요 언론
* 링크 없는 데이터는 원칙적으로 제외하거나 신뢰도 낮음 표시

## 3. 알림 형식

새로운 데이터가 감지되면 나에게 보내는 메시지는 반드시 아래 구조를 따라라.

### 필수 출력 형식

1. **제목**
2. **발생 시각 (KST)**
3. **카테고리**

   * 예: Circle / Tether / Regulation / Bill / SEC / Stablecoin Market / Partner
4. **핵심 요약**

   * 3줄 이내
5. **상세 설명**

   * 무슨 일이 발생했는지
   * 누가 발표했는지
   * 왜 중요한지
   * 기존 상황 대비 무엇이 달라졌는지
6. **주가 및 Circle 영향 분석**

   * Circle 기업가치/사업/심리에 미칠 영향
   * 단기/중기 관점 구분 가능하면 구분
   * 호재 / 악재 / 중립 판단
   * 왜 그런 판단인지 논리적으로 설명
7. **관련 링크**

   * 원문 링크들 나열
8. **중복 여부 체크 결과**

   * 왜 이것이 "새로운 데이터"라고 판단했는지 간단히 기록

### 예시 포맷

[새 알림] {제목}

* 발생 시각(KST): {시간}
* 카테고리: {카테고리}

1. 핵심 요약

* ...
* ...
* ...

2. 상세 설명

* ...
* ...
* ...

3. 주가 및 Circle 영향 분석

* 방향성: 호재 / 악재 / 중립
* 단기 영향: ...
* 중기 영향: ...
* 근거: ...

4. 관련 링크

* ...
* ...
* ...

5. 신규성 판정

* 이전에 저장된 이벤트 DB와 비교한 결과, {어떤 점이 새로움인지 설명}

## 4. 기술 요구사항

### 개발 원칙

* 유지보수 쉽게 구조화
* 실제 서비스처럼 동작하도록 작성
* 에러 발생 시 죽지 말고 계속 재시도
* 로그 남기기
* 중복 전송 방지
* 소스별 신뢰도 관리
* 새 데이터 판정 로직 포함

### 반드시 필요한 기능

1. **주기적 수집**

   * 뉴스 / 공시 / 공식 발표 / 규제 사이트 / RSS / 웹페이지 등 주기적 확인

2. **중복 제거**

   * URL 중복 제거
   * 제목 유사도
   * 본문 핵심 유사도
   * 동일 사건 클러스터링
   * 이미 보낸 이벤트 DB 저장

3. **신규성 판정**

   * 이전 이벤트와 비교해 truly new인지 판정
   * 단순 재게시 제외
   * 기존 기사와 동일 사실이면 제외
   * 새 수치/새 발표/새 문서/새 표결/새 규제조치/새 소송 단계 등은 신규로 인정 가능

4. **요약/분석 생성**

   * 각 이벤트마다 핵심 요약
   * 상세 설명
   * Circle 및 주가 영향 분석 자동 생성

5. **알림 전송**

   * 내가 바로 받을 수 있는 방식으로 구현
   * 우선순위 예시: Telegram / Discord webhook / 이메일 / Slack 중 하나 이상
   * 알림 수신 설정은 config 파일에서 바꿀 수 있게

6. **Google Cloud VM 상시 실행**

   * Ubuntu 환경 기준
   * PM2 또는 systemd로 24시간 실행
   * 재부팅 후 자동 실행
   * 설치 방법과 배포 방법까지 문서화

## 6. 구현 요구

### 원하는 산출물

아래를 모두 만들어라.

1. **실행 가능한 전체 프로젝트 코드**
2. **README.md**

   * 설치법
   * 환경변수 설정법
   * Google Cloud VM 배포법
   * PM2 또는 systemd 실행법
3. **config 파일 예시**
4. **중복 이벤트 저장용 DB**

   * sqlite 추천
5. **로그 시스템**
6. **에러 처리 및 재시도 로직**
7. **테스트 가능한 구조**
8. **소스 추가/수정 쉽게 모듈화**

### 선호 스택

* Python 우선
* requests / feedparser / beautifulsoup4 / sqlite3 등 활용 가능
* 필요하면 playwright 또는 selenium 일부 사용 가능하나 과도한 복잡성은 피하라
* LLM API가 필요하면 붙일 수 있게 추상화
* 기본은 비용 적게 들도록 설계

## 7. 데이터 판정 로직을 특히 신경 써라

내가 가장 중요하게 보는 것은 **“이전 뉴스 제외, 진짜 새로운 정보만 알림”** 이다.

따라서 아래를 꼭 구현하라:

* 최근 전송된 이벤트 저장
* 제목 유사도 비교
* URL canonicalization
* 동일 사건 묶기
* 본문 핵심 문장 비교
* 이미 알려진 사건인지 판단
* 기존 사건의 후속 업데이트인지 판단
* 후속 업데이트라도 **새로운 사실이 없으면 알리지 않기**
* 새로운 문서 번호, 새로운 규제조치, 새로운 법안 단계, 새로운 수치, 새로운 파트너십, 새로운 소송 단계 등은 별도 신규 이벤트로 허용

## 8. 개발 결과 출력 방식

한 번에 끝내지 말고, 아래 순서로 진행해라.

### 1단계

먼저 전체 아키텍처를 제시해라.

* 어떤 파일들이 필요한지
* 데이터 흐름이 어떤지
* 신규성 판정 로직을 어떻게 설계할지

### 2단계

그다음 실제 코드 파일들을 생성해라.

* 파일별로 코드 전부 작성

### 3단계

그다음 배포 방법을 작성해라.

* Google Cloud VM Ubuntu 기준
* 패키지 설치
* git clone 또는 파일 업로드
* venv 설정
* 실행
* PM2/systemd 등록
* 자동 재시작

### 4단계

마지막으로 테스트 방법까지 제시해라.

* 새 기사 감지 테스트
* 중복 제거 테스트
* 알림 전송 테스트

## 9. 추가 지시

* 절대 대충 만들지 마라
* 실사용 가능한 수준으로 만들어라
* 추상적인 설명만 하지 말고 실제 코드까지 작성해라
* 내가 바로 복붙해서 실행 가능한 형태를 선호한다
* 환경변수명, 폴더 구조, 실행 명령어까지 구체적으로 작성해라
* 코드 안 주석은 충분히 달아라
* Windows에서 작성 후 Google Cloud Ubuntu에 올려 실행하는 흐름도 고려해라

이제 위 요구사항을 만족하는 프로젝트를 설계하고 작성해라.
